SIMD.js value semantics proposal

Introduction

This proposal adds SIMD types and operations to Javascript, focusing on the value semantics. The proposal adds new primitive types float32x4, etc, together with wrappers and a definition of their behavior in the language. The proposal discusses float32x4 and just a couple functions on it, with the idea that it would extend similarly to other SIMD types. Rough definitions for more types and functions that operate on them can be found at the SIMD.js spec. This document is organized in terms of where changes would be made to the ES6 spec.

The key problem that this spec aims to solve, over the previous informal SIMD specification, is the way that SIMD values were previously objects with identity-based equality. Maintaining object identity puts a big burden on compilers to maintain this identity through operations, where they would rather be able to duplicate and de-duplicate SIMD values arbitrarily based on algebraic identities. By making SIMD values into primitive types with structural equality, compilers are given more freedom. However, this gives a lot to expand on in the spec.

Ideally, SIMD values will fit into a larger value types proposal. Such a proposal would be a bit more involved, but good work has already been done in that direction. This document describes SIMD without a larger value type system, but it aims to be consistent with how value types might work, and once someone steps forward to describe value types in more detail, it will be great to simplify this text by just explaining SIMD in terms of value types.

Many parts of this spec edit existing parts of the ES6 spec. Although ecmarkup generates numbering at the beginning of headers, these won't correspond to the numbering within the existing ECMA spec, so I've included a matching numbering in parentheses afterwards, referring to the ES6 spec. I also intend to add crossreferencing links where appropriate.

SIMD types (6.1.8)

Float32x4 (6.1.8.1)

Float32x4 is a SIMD type representing four 32-bit floating point values. Float32x4 values can be created using the [[Call]] operation on the SIMD.Float32x4 object, described in 27.1.1.

ToPrimitive ( input [, PreferredType] ) (7.1.1)

Argument type: Float32x4
Result: return input An additional option considered is to define this as calling ToObject on it, and then ToPrimitive on the resulting wrapper object. SIMD objects (27)

The SIMD global object has several constructor properties, one for each SIMD type

Float32x4 objects (27.1)

As with Boolean, String, etc, this is a constructor for the wrapper when invoked with new, and returns a primitive when called as a function.

The Float32x4 constructor (27.1.1)

The Float32x4 constructor reachable from the global object as SIMD.Float32x4. It is sometimes referred to in this spec as %SIMDFloat32x4% to reduce ambiguity and emphasize that, even if the SIMD object is modified or shadowed, references in this spec refer to the original binding.

Float32x4( value ) (27.1.1.1)

This description applies if Float32x4 is called with exactly one argument. 1. If NewTarget is undefined, throw a ReferenceError (NB: TypeError?) 1. If value is not of the type Float32x4, throw a TypeError 1. Let O be OrdinaryCreateFromConstructor(NewTarget, "%SIMDFloat32x4Prototype%", «[[Float32x4Data]]» ). 1. ReturnIfAbrupt(O). 1. Set the value of O’s [[Float32x4Data]] internal slot to n. 1. Return O.

Float32x4( x, y, z, w ) (27.1.1.2)

This description applies if Float32x4 is called with exactly four arguments. If it is called with a different number of arguments from one or four, throw a TypeError (NB: Or, call this version, with some things being undefined and/or some values ignored?) 1. Let xf be ToFloat32(x) 1. ReturnIfAbrupt(xf) 1. Let yf be ToFloat32(y) 1. ReturnIfAbrupt(yf) 1. Let xf be ToFloat32(z) 1. ReturnIfAbrupt(zf) 1. Let yf be ToFloat32(w) 1. ReturnIfAbrupt(wf) 1. Return a Float32x4 primitive with the internal slots { [[Float32x4X]]: xf, [[Float32x4Y]]: yf, [[Float32x4Z]]: zf, [[Float32x4W]]: wf}

SIMD.Float32x4.extractLane(t, i) (27.1.1.3)

1. If Type(t) is not Float32x4, throw a TypeError exception 1. Let index be ToNumber(i) 1. If i is 0, return t.[[Float32x4X]] 1. If i is 1, return t.[[Float32x4Y]] 1. If i is 2, return t.[[Float32x4Z]] 1. If i is 3, return t.[[Float32x4W]] 1. Else, return ReferenceError

SIMD.Float32x4.add(a, b) (27.1.1.3)

1. If Type(a) is not Float32x4 or Type(b) is not Float32x4, throw a TypeError exception 1. Return a Float32 with the internal slots {[[Float32x4X]]: ToFloat32(a.[[Float32x4X]] + b.[[Float32x4X]]), [[Float32x4Y]]: ToFloat32(a.[[Float32x4Y]] + b.[[Float32x4Y]]), [[Float32x4Z]]: ToFloat32(a.[[Float32x4Z]] + b.[[Float32x4Z]]), [[Float32x4W]]: ToFloat32(a.[[Float32x4W]] + b.[[Float32x4W]])}

The SIMD.Float32x4.prototype (27.1.2)

Previously, if accessors like x were included, these would be spec'd as getters (with null setter) on the SIMD.Float32x4.prototype . However, they have been removed in favor of SIMD.Float32x4.extractLane. Are there any non-trivial properties? Users could add methods like .add() here if they want to use SIMD in an object-oriented way.
